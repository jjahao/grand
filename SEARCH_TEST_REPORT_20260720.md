## 3. 已確認事實

### 3.1 Shop2000 原生搜尋把空格當片語，不是 AND
以下為直接打搜尋 URL 的線上實測結果：

| 查詢字 | URL | 結果 |
|---|---|---|
| `醬油` | `/product?kw=醬油` | `共 77 筆` |
| `減鹽` | `/product?kw=減鹽` | `共 10 筆` |
| `減塩` | `/product?kw=減塩` | `共 7 筆` |
| `醬油 減鹽` | `/product?kw=醬油%20減鹽` | `查無商品` |
| `醤油 減塩` | `/product?kw=醤油%20減塩` | `查無商品` |
| `昆布 醬油` | `/product?kw=昆布%20醬油` | `查無商品` |
| `醬油 昆布` | `/product?kw=醬油%20昆布` | `查無商品` |

**結論**：只要把兩個關鍵字直接一起丟給 Shop2000 原生 `kw=`，它就會把空格視為整段字串，結果直接變成 0 筆。這是平台行為，不是我們皮膚可改的地方。

### 3.2 前台測試會員可登入，但沒有足夠分類權限
- 使用 keychain 內測試會員帳密登入前台成功
- 但進受限分類（例如 `https://grand.shop2000.com.tw/product/654002/1365721`）仍看到會員牆
- 表示**這個測試會員沒有足夠前台分類權限**

**影響**：本輪無法用該帳號完整驗證「會員限定分類內」的最終搜尋體驗，只能驗證公開/可見搜尋頁與程式碼。

### 3.3 現行 `shop-skin.js` 已經有三層搜尋強化，不是單純原生搜尋
在 `/Volumes/AI_HQ/grand_store/shop-skin.js` 已確認：

1. **簡體→繁體正規化**
   - `GP_S2T` / `gpS2T()` / `gpNorm()`
   - 例如：`酱油` → `醬油`

2. **品牌/通路同義詞**
   - `GP_SYN` / `gpSynMatch()`
   - 例如：`好市多` ↔ `costco`、`7-11` ↔ `711` ↔ `seven eleven`

3. **多關鍵字 AND + 多頁聚合**
   - `gpFilter` / `gpTokenize()` / `gpApplyFilter()`
   - `gpExpandMultiPage()` 會用 `to_p=` 背景抓原生其他分頁，最多 15 頁，再合併後做 AND 過濾

### 3.4 目前理論設計與我先前口頭描述不一致，應以程式碼為準
本次重新核對後，**目前線上皮膚不是只有「第一頁 + 第一個關鍵字」**。
它在原始碼裡已經有：
- 第一個關鍵字先餵給 Shop2000 原生搜尋
- 完整查詢透過 `gpq=` 帶回前台
- 前台再做 AND 過濾
- 並嘗試抓 `to_p=2...15` 的其他原生頁面合併

所以接下來真正要驗證的，不是「有沒有做」，而是：
- `to_p=` 在實站是否穩定可用
- 某些分類/搜尋頁是否根本沒成功套皮
- 某些商品名稱是否仍因字詞差異而漏搜

## 4. 真正剩下的問題定義

### 問題 A：某些頁面/狀態下，皮膚搜尋 UI 沒有成功接管
本輪在 Chrome 實測時，部分搜尋結果頁雖然 script tag 已載入：
- `https://jjahao.github.io/grand/shop-skin.js?v=4`

但頁面上沒有出現：
- `#gp-wrap`
- `#gp-kw`
- `.gp-card`

這代表至少在這組前台 session / 某些頁面狀態下，**皮膚有載入 script，但沒有成功 render 出搜尋 UI**。

這是高優先級，因為如果皮膚沒接管，後面的多關鍵字、同義詞、多頁聚合全部不會生效。

### 問題 B：前台會員驗證環境不足
測試會員可以登入，但受限分類仍沒有權限。
這使得「會員限定分類」搜尋測試無法完整驗證。

### 問題 C：就算皮膚成功接管，現有字典仍可能不夠
雖然已有簡繁轉換與部分同義詞，但還可能漏掉：
- `減鹽` ↔ `減塩`
- `醬油` ↔ `醤油`
- `昆布` ↔ `こんぶ` / `コンブ`
- `毛巾` ↔ `タオル`
- `洗髮` ↔ `シャンプー`

這類跨語言/跨書寫系統對照，目前沒有完整覆蓋。

## 5. 可執行改善方案（依風險排序）

### 方案 1：先做「皮膚接管成功率」診斷（最優先）
**目標**：先確認為什麼有些頁面 script 已載入，但 `gp-wrap` 沒 render。

**做法**：
- 在 `shop-skin.js` 加輕量 console 診斷點：
  - 是否進入 `run()`
  - `isProductListPage()` 結果
  - `gatherProducts()` / `gpParseDoc(document)` 拿到幾筆
  - `buildPrettyList()` 是否被呼叫
  - `gpExpandMultiPage()` 抓到幾頁、每頁幾筆
- 先只做 log，不改搜尋邏輯

**優點**：
- 風險最低
- 可以把現在「到底哪裡沒生效」直接抓出來
- 不會誤改已上線邏輯

**成本 / 速度**：
- 低
- 1 輪即可得到答案

### 方案 2：補強日中漢字 / 常見詞對照表（安全、快）
**目標**：提高命中率，不碰平台分頁機制。

**做法**：
- 擴充 `GP_SYN` 和正規化規則
- 優先加高頻商品詞：
  - `鹽 ↔ 塩`
  - `醬 ↔ 醤`
  - `麵 ↔ 麺`
  - `豬 ↔ 豚`
  - `毛巾 ↔ タオル`
  - `洗髮 ↔ シャンプー`
  - `沐浴乳 ↔ ボディソープ`
  - `保溫瓶 ↔ 水筒`

**優點**：
- 純前端比對
- 不增加對 Shop2000 的請求壓力
- 可逐批擴充、好驗收

**風險**：
- 低
- 主要是字典維護成本

### 方案 3：把 brief / 副標 / 隱藏輔助欄位納入搜尋（安全、中效）
**目標**：商品名之外再增加一層命中來源。

**做法**：
- 目前 `gpApplyFilter()` 只對 `p.name` 搜
- 可改成：`p.name + brief + categoryLabel + synonyms` 一起比對
- 對客人看不到，但對搜尋有效

**優點**：
- 對既有商品資料利用率高
- 對跨語言補字很有幫助

**風險**：
- 低到中
- 需要先確認哪些欄位目前已穩定抓到

### 方案 4：建立「搜尋別名索引」欄位（最確實，但要動資料）
**目標**：從根本提升商品可搜性。

**做法**：
- 每個商品建立一組 `search_alias`
- 內容可含：
  - 中文名
  - 日文名
  - 品牌英文
  - 類別詞
  - 常用別稱
- 前台搜尋時直接對 alias blob 做比對

**優點**：
- 命中率最高
- 規模化後最穩

**風險**：
- 中
- 需要改資料生成流程或商品 enrich 流程
- 屬於架構級調整，不能偷改

## 6. 我建議的實際落地順序

### 第一階段（安全、最快）
1. 先做 **方案 1：皮膚接管診斷**
2. 同時做 **方案 2：擴充高頻詞對照**

### 第二階段（穩定補強）
3. 做 **方案 3：brief / 類別詞納入搜尋**

### 第三階段（若 James 要更高命中率）
4. 再評估 **方案 4：商品搜尋別名索引**

## 7. 本輪結論
- Shop2000 原生搜尋本身不支援空格多關鍵字 AND，這點已線上實測確認
- 現行 `shop-skin.js` 實際上已經有：
  - 簡繁正規化
  - 同義詞
  - 多關鍵字 AND
  - 多頁聚合
- 因此目前最值得先查的，不是再加新功能，而是：
  1. **皮膚是否在所有搜尋頁都成功接管**
  2. **多頁聚合在實站是否真的穩定跑完**
  3. **詞典覆蓋率是否不足**

## 8. 建議審批選項
- **A. 只做診斷**：先查清楚皮膚為何有時不接管，不碰搜尋邏輯
- **B. 診斷 + 詞典補強**：在 A 基礎上，同步加一批高頻同義詞
- **C. 診斷 + 詞典 + brief 搜尋**：中等改動，提升命中率
- **D. 直接做搜尋別名索引規劃**：最大工程，先出 spec 再動手
